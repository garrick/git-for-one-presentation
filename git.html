What is Git?

Git is:

  • "A distributed revision control system" and...
  • "A source code management system (SCM)" (if you believe Wikipedia)
  • Self described in the man page as...
  • git - the stupid content tracker

The history of Git

How Git got started:

  • Began when Linus Torvalds & other Linux kernel contributors got fed up with political issues around using non-open source software (called BitKeeper) to maintain the Linux kernel.
  • Linus & company decided they were better at solving technical problems than political ones.
  • From the start in early 2005 through late 2005 with the 1.0 release, Git went from prototype to meeting it's three primary goals inspired by BitKeeper and Monotone which were:
  • Take CVS as an example of what not to do; if in doubt, make the exact opposite decision.
  • Support a distributed, BitKeeper-like workflow.
  • Very strong safeguards against corruption, either accidental or malicious.

Advantages of Git

Git is good for:

  • Tracking source control changes from multiple contributors
  • Dealing with multiple branches of source code
  • Supporting software development without a centeralized server (e.g. like Subversion)

Disadvantages of Git

Git has it's downfalls. Such as:

  • Git is not very user friendly
  • There are some GUI tools, but they are best used as a visual enhancement to the command line utilities
  • Git's promiscuous branching model can discourage true continuous integration

Why bother to use Git?

  • It's pretty simple for individual projects
  • It requires no maintined server
  • You can also use it to "deploy" on $5/month hosting accounts
  • Resumé fodder: Dice.com shows almost the same number of postings for Git as Subversion
  • Think Different: because Git is different from Subversion, it gives you a different perspective on the nature of maintaining & sharing source code

Less talk, dude. More code.

I've already installed Git & I'll be demoing live on the command line. We'll start with creating a project directory and initializing a Git repository.

        mkdir hello_world
        cd hello_world
        git init .

Git responds telling us the command was successful:

        Initialized empty Git repository in /home/badlogin/hello_world/.git/

What's going on here?

One of the most common Git commands is to ask what's going on with 'status' like so:

git status

git status says nothings going on:

# On branch master
#
# Initial commit
#
nothing to commit (create/copy files and use "git add" to track)

This is a command you'll use frequently to understand the state of your workspace.

Let's do some project set-up

Remember - Git tracks content, really. Not "source files" or "directories" or whatever. So let's give it some content to track. Like a .gitignore file:

gvim .gitignore
git status

git status indicates an untracked file:

# On branch master
#
# Initial commit
#
# Untracked files:
#   (use "git add ..." to include in what will be committed)
#
#	.gitignore
nothing added to commit but untracked files present (use "git add" to track)

We need to tell Git what to track

Git has seen a content change, but doesn't track that object. So we have to tell it to explicitly.

git add .gitignore
git status

git status indicates we have changes in the index:

# On branch master
#
# Initial commit
#
# Changes to be committed:
#   (use "git rm --cached ..." to unstage)
#
#	new file:   .gitignore
#

Now, let's actually commit to the repo

We've told Git to add our .gitignore changes to the index, but we haven't told git to store the change as a revision. That's kind of the whole point of this, so let's do that.

git commit -m "Initial commit."

Now we see the kind of command where Git says something without us having to ask for status.

[master (root-commit) 4c86a32] Initial commit
 1 file changed, 1 insertion(+)
 create mode 100644 .gitignore

And just where are we now?

Ok. That did something, as you might expect we'll ask Git the status of our workspace. Just to be sure.

git status

And all is quiet.

# On branch master
nothing to commit, working directory clean

Let's make a change to an existing file

Now that the file is tracked by Git, let's make a change to it's content then check our workspace status. 

gvim .gitignore
git status

Yep. Git knows about .gitignore now 

# On branch master
# Changes not staged for commit:
#   (use "git add ..." to update what will be committed)
#   (use "git checkout -- ..." to discard changes in working directory)
#
#	modified:   .gitignore
#
no changes added to commit (use "git add" and/or "git commit -a")

We can see our differences between workspace and index

Git has powerfull change 'diffing' built in. 

git diff .gitignore

You'll eventually become comfortable with diff ouput format as shown below. To learn more, Google is your friend. 

diff --git a/.gitignore b/.gitignore
index b25c15b..76b2cbc 100644
--- a/.gitignore
+++ b/.gitignore
@@ -1 +1,2 @@
 *~
+*.swp

So we add our new change to the index

This is just like the earlier 'add'. We're adding a change, not a file! 

git add .gitignore
git status

Now, our change is in the index and ready to be committed. 

# On branch master
# Changes to be committed:
#   (use "git reset HEAD ..." to unstage)
#
#	modified:   .gitignore
#

And finally, we commit our change

Git 

git commit -m "Telling Git to ignore Vim swap files"

Again, we get the confirmation that we made a change to the repo. 

[master b46572c] Telling Git to ignore Vim swap files
 1 file changed, 1 insertion(+)

You can view your commit logs

Over time on a project, checking log history can be useful. 

git log

From this, you can diff on specific commits (hint: you only need the first 6 characters of each commit. Not the whole SHA1!)

Git solo project workflow

mkdir some_project        #Step1 - create a workspace directory
cd some_project           #Step2 - get ready to issue Git commands there
git init .                #Step3 - initialize workspace to hold a .git repo
gvim somefile.txt         #Step4 - Start working
git add somefile.txt      #Step5 - Add your project changes to the index
git commit -m "I coded"   #Step6 - Commit at regular intervals

This is hepful even for small pieces of work.

Git EZ Archiving

git archive --format zip --output /full/path/to/zipfile.zip master

Your Git project will have a .git directory. If you want to make a copy of the current revision of your repo without including your full revision history, the git archive command is quite handy. The above command works for the default 'master' branch. See the man page for a world of options.

Accessing open source projects via Git

git clone is the simplest way to access a Git project. 

git clone https://github.com/michalbe/Simple-game-with-HTML5-Canvas.git

Most open source projects will have a read-only Git URL. The git clone command let's you copy all of the repository content onto your own system.

Keeping up with open source projects via Git

git pull will bring down the latest content.

git pull

Note: if you've made conflicting local changes, this may not work easily. This is where more hands on work with Git will pay off. Or, you can just blow away the directory and pull down a new copy...

Setting up a remote SSH repo for Backup & Deploy

On your remote system:

ssh your_user@yourdomain.com
mkdir hello_world.git
cd hello_world.git/
git init --bare
exit

Now, in your local Git repo do: 

git remote add origin your_user@yourdomain.com:/home/your_user/hello_world.git
git push origin master

This is now your remote origin of your repo, the authority of your project. With this set up, you can now easily clone and deploy your project.

Simple deploy on the big bad intertubes

On your remote system:

ssh your_user@yourdomain.com
cd public_html #or wherever you deploy to
git clone ~/hello_world.git

Be warned, witout some .htaccess restrictions, the contents of your .git repo are accessible as well! When you want to update your deployment after you've pushed into your origin repo, log into your hosting account and simply do a git pull in your server side clone:

ssh your_user@yourdomain.com
cd public_html/hello_world 
git pull --rebase
